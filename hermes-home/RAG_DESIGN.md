# RAG-дизайн для Hermes помощника по Element

## 1. Цель

Сделать помощника, который отвечает **по вашему внедрению** Element, а не по абстрактной документации из интернета.

## 2. Какие знания индексировать

### Контур A — локальные документы (высший приоритет)
1. FAQ для сотрудников.
2. Инструкция по входу и первичной настройке.
3. Политика использования служебного мессенджера.
4. Памятка по мобильному клиенту.
5. Памятка по desktop/web клиенту.
6. Порядок эскалации в ИТ.
7. Памятка по шифрованию, верификации устройств и ключам.
8. Список типовых ошибок и решений.

### Контур B — operational knowledge
1. Разрешённые клиенты и версии.
2. SSO/LDAP/2FA/VPN-особенности.
3. Ограничения по вложениям, звонкам, гостевому доступу.
4. Политика именования пространств/комнат, если пользователю это нужно знать.
5. Контакты поддержки, режим работы, SLA если допустимо раскрытие.

### Контур C — внешняя документация
1. Официальные docs Element.
2. Официальные docs Matrix только на базовом пользовательском уровне.

## 3. Что не класть в индекс без фильтра

- секреты;
- токены;
- внутренние адреса админки;
- лишние инфраструктурные детали;
- runbook'и для администраторов, если агент ориентирован на сотрудников;
- устаревшие версии инструкций без маркировки актуальности;
- сырые шаблонные маркеры и служебные placeholder-имена (`SUPPORT_CONTACT`, `SECURITY_CONTACT`, `SUPPORT_CHANNEL`, `ELEMENT_WEB_URL` и т.п.) в user-facing knowledge base.

## 4. Единица индексации

Оптимально хранить не огромные документы, а небольшие chunks:
- 400–900 слов;
- 1 тема на chunk;
- обязательные метаданные у каждого куска.

## 5. Метаданные chunk'ов

Рекомендуемые поля:
- `source_id`
- `title`
- `doc_type` — faq / policy / howto / troubleshooting / escalation
- `product_area` — login / notifications / encryption / rooms / files / mobile / desktop
- `audience` — employee / manager / support
- `platform` — web / desktop / ios / android / all
- `criticality` — normal / warning / risky
- `version`
- `updated_at`
- `authoritative` — true/false

## 6. Retrieval strategy

### Базовый pipeline
1. Классифицировать вопрос по интенту.
2. Определить платформу: web / desktop / mobile.
3. Подтянуть 3–6 лучших chunk'ов из локальной базы.
4. Отфильтровать по свежести и authoritative=true.
5. Если локальных знаний мало — добавить 1–2 внешних chunk'а из официальной документации.
6. Сгенерировать ответ только на основе выбранного контекста.

### Полезные интенты
- вход / логин
- настройка профиля
- уведомления
- комнаты и пространства
- треды и упоминания
- файлы и медиа
- звонки
- устройства и сессии
- верификация устройства
- ключи шифрования
- восстановление доступа
- ошибка / troubleshooting
- правила использования
- эскалация в поддержку

## 7. Answer policy поверх RAG

Если найден authoritative локальный chunk — он имеет приоритет.
Если найден только внешний chunk — нужно маркировать ответ как общий и рекомендовать свериться с локальной инструкцией, если тема чувствительная.
Если retrieval пустой — не фантазировать, а переводить в режим уточнения/эскалации.

## 9. Минимальный набор тестовых вопросов

1. Как войти в Element впервые?
2. Как настроить уведомления только на упоминания?
3. Почему не видно старые сообщения на новом телефоне?
4. Как подтвердить новое устройство?
5. Можно ли использовать личный смартфон?
6. Что делать, если не приходит код/SSO не работает?
7. Где искать рабочие пространства и комнаты?
8. Как отправить файл и есть ли ограничения?
9. Что делать при подозрении на взлом аккаунта?
10. Куда писать, если инструкция не помогла?

## 10. Режимы зрелости

### MVP
- markdown-файлы;
- простой embedding/vector store;
- ручное пополнение FAQ;
- 20–30 проверенных вопросов.

### Next
- интент-классификация;
- reranking;
- контроль версий документов;
- флаг устаревания инструкций;
- аналитика unanswered questions.

### Mature
- автоматическая загрузка из wiki/Confluence/Notion/Git;
- approval workflow для публикации знаний;
- разделение user-facing и admin-only контуров;
- обратная связь пользователей по полезности ответа.
