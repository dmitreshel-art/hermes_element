# Hermes Element Helpdesk Assistant

AI-помощник для пользователей служебного мессенджера Element/Matrix.
Развёртывается как изолированный Docker-контейнер с отдельным профилем Hermes.

## Возможности

- Ответы на типовые вопросы пользователей Element (вход, уведомления, комнаты, шифрование, устройства)
- База знаний из 30+ документов с общими инструкциями по Element/Matrix
- Локальная база знаний уже заполнена под ваш контур; при необходимости её можно дальше уточнять и расширять
- Ограниченный набор инструментов: только чтение KB + memory + skills + session_search
- Самообучение: агент запоминает эффективные формулировки ответов и улучшает их со временем

## Быстрый старт

### 1. Клонируйте репозиторий

```bash
git clone https://github.com/dmitreshel-art/hermes_element.git
cd hermes_element
```

### 2. Подготовьте `.env`

Скопируйте пример и заполните реальные значения:

```bash
cp hermes-home/.env.example hermes-home/.env
```

Минимально обязательные переменные:

| Переменная | Описание |
|---|---|
| `MATRIX_HOMESERVER` | URL вашего Matrix-сервера (например `https://matrix.example.org`) |
| `MATRIX_USER_ID` | Matrix ID бота (например `@helpdesk:example.org`) |
| `MATRIX_PASSWORD` или `MATRIX_ACCESS_TOKEN` | Пароль бота или access token |
| `MATRIX_ALLOWED_USERS` | Список разрешённых пользователей (через запятую) или `*` для всех |
| `OPENAI_API_KEY` | Ключ к OpenAI-совместимому API |

### 3. Проверьте локальную базу знаний

Базовые значения для вашего контура уже подставлены в KB: адрес сервера, вход, разрешённые клиенты, поддержка через этого бота, правила по комнатам и инструкции по устройствам/шифрованию.

Перед live-запуском стоит только дополнительно решить, нужны ли вам более строгие локальные регламенты по:

```text
- массовым упоминаниям
- типам данных и чувствительным файлам
- режиму работы специалистов
```

### 4. Выполните предстартовую проверку

```bash
docker compose -f docker/docker-compose.yml config
```

Если команда проходит без ошибок, можно запускать контейнер.

### 5. Запустите Docker

```bash
docker compose -f docker/docker-compose.yml up -d
```

### 6. Проверьте запуск

```bash
docker compose -f docker/docker-compose.yml logs -f
```

Ожидаемый результат:
- контейнер стартует без ошибок;
- entrypoint не ругается на отсутствие `.env` или placeholder-значения;
- Hermes gateway запускается в foreground-режиме;
- бот подключается к Matrix и отвечает в DM/разрешённых комнатах.

## Безопасность

### Активные toolsets

| Toolset | Статус | Назначение |
|---|---|---|
| `file` | ✅ Включён | Чтение файлов базы знаний |
| `memory` | ✅ Включён | Память между сессиями |
| `skills` | ✅ Включён | Самообучение (улучшение ответов) |
| `session_search` | ✅ Включён | Поиск по прошлым диалогам |

### Отключённые toolsets

| Toolset | Статус | Почему отключен |
|---|---|---|
| `terminal` | ❌ | Нет доступа к оболочке |
| `browser` | ❌ | Нет доступа к браузеру |
| `web` | ❌ | Нет доступа к вебу |
| `code_execution` | ❌ | Нет выполнения кода |
| `vision` | ❌ | Нет анализа изображений |
| `image_gen` | ❌ | Нет генерации изображений |
| `tts` | ❌ | Нет генерации речи |
| `messaging` | ❌ | Нет отправки сообщений напрямую |
| `homeassistant` | ❌ | Нет доступа к умному дому |
| `todo` | ❌ | Нет управления задачами |
| `clarify` | ❌ | Минимизация интерактивных ветвлений |
| `cronjob` | ❌ | Нет планирования задач |
| `delegation` | ❌ | Нет делегирования подзадач |

Ограничения применены **механически** в `config.yaml`, не только через системный промпт.

## Политика памяти

### Что агент хранит в пользовательской памяти
- Уровень навыков пользователя (новичок / опытный)
- Предпочитаемый клиент (web / desktop / iOS / Android)
- Повторяющиеся затруднения (например, «путается в уведомлениях»)
- Предпочитаемый стиль ответа (подробный / краткий)

### Что агент НЕ хранит
- Пароли, recovery-ключи, токены
- Содержание сообщений
- Персональные данные
- Секреты

### Что агент хранит в skills
- Улучшенные формулировки ответов
- Mini-playbook'и для повторяющихся сценариев
- Безопасные формулировки для сложных тем (ключи, шифрование)

## Кастомизация

### Добавление локальных документов

Положите ваши внутренние документы в `hermes-home/knowledge/`:
- Инструкция по первому входу
- Правила использования служебного мессенджера
- Контакты ИТ-поддержки
- Политика по файлам/звонкам/гостям

### Добавление FAQ

Создайте файл `hermes-home/knowledge/faq-YOUR-TOPIC.md` по шаблону:

```markdown
# FAQ: Название темы

## В: Вопрос?
**А:** Ответ.

## В: Другой вопрос?
**А:** Другой ответ.
```

### Использование другого LLM-провайдера

По умолчанию в `hermes-home/config.yaml` выбран `openai / gpt-4o-mini`, потому что это проще всего для portable deployment через `OPENAI_API_KEY`.

Если хотите использовать другой провайдер:
1. отредактируйте `hermes-home/config.yaml`;
2. замените `model.provider` и `model.default`;
3. укажите соответствующий API-ключ в `hermes-home/.env`.

## Перенос на другой сервер

1. Клонируйте этот репозиторий на целевой сервер
2. Скопируйте `hermes-home/.env.example` в `hermes-home/.env`
3. Заполните реальные Matrix и LLM credentials
4. Проверьте `docker compose -f docker/docker-compose.yml config`
5. Запустите `docker compose -f docker/docker-compose.yml up -d`
6. Проверьте логи через `docker compose -f docker/docker-compose.yml logs -f`

## Результаты предстартовой ревизии deploy-конфига

В репозитории были найдены и исправлены такие проблемы:

1. **Неверный Docker CMD**
   - было: `gateway`
   - стало: `gateway run`
   - иначе контейнер мог запускаться не в том режиме.

2. **`.env` не монтировался в контейнер как файл**
   - entrypoint проверял `/home/hermes/.env`, но compose передавал только env vars
   - теперь весь `hermes-home/` монтируется в `/home/hermes`, поэтому `.env`, `config.yaml`, KB и логи видны контейнеру как реальные файлы.

3. **`docker compose config` падал без `.env`**
   - теперь `env_file` сделан optional (`required: false`)
   - это позволяет валидировать compose ещё до создания боевого `.env`.

4. **В Dockerfile была неверная стратегия установки Hermes**
   - раньше предполагалось, что `hermes-agent==0.10.0` доступен через PyPI, но сборка падала с `No matching distribution found`
   - теперь Hermes ставится напрямую из официального GitHub-репозитория `NousResearch/hermes-agent`, и сборка проходит успешно.

5. **Не хватало Matrix runtime dependencies**
   - добавлены `libolm-dev`, `libffi-dev`, `pkg-config`, `mautrix`, `python-olm`.

6. **Был конфликт между модельным провайдером и `.env.example`**
   - раньше конфиг ожидал `openai-codex`, а `.env.example` подсказывал `OPENAI_API_KEY`
   - теперь по умолчанию выбран `openai / gpt-4o-mini`, что согласовано с `.env.example`.

7. **Ограничения toolsets были заданы только для CLI**
   - теперь те же safe toolsets явно заданы и для платформы `matrix`.

8. **В config оставался placeholder room id**
   - удалён example-room из `free_response_rooms`, теперь список пустой и безопасный по умолчанию.

9. **Entrypoint не останавливал контейнер при placeholder-секретах**
   - теперь запуск прерывается, если в `.env` остались `CHANGE_ME`, `***`, `matrix.example.org` и т.п.

### Итог
После ревизии проект стал заметно более готовым к реальному переносу и запуску:
- меньше скрытых сюрпризов при первом `docker compose up`;
- понятнее связь между `.env`, config и runtime;
- меньше риска запустить контейнер с пустыми или фиктивными значениями.
